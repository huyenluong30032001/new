01 - INTRODUCTION

Meal planning can often feel like a daunting task, especially for individuals juggling busy schedules or those new to cooking. With this in mind, I set out to create BiteBuddy MealPlan , a user-focused web application designed to simplify meal planning while making it a fun and engaging experience. This project aimed to empower users to explore recipes, plan meals efficiently, and connect with a community of like-minded food enthusiasts.

Through BiteBuddy, I explored how intuitive design, seamless API integration, and personalized user experiences could come together to address the everyday challenges of meal planning. From the initial research phase to developing wireframes, creating high-fidelity mockups, and finally coding the platform, I followed the design thinking process to ensure every feature was user-centered. The result is a platform that combines functionality with creativity, offering users a dynamic and supportive environment for their culinary journeys.

03 - Define

Key Focus Areas

Overwhelming Meal Planning Tools

Many users, particularly beginners, feel overwhelmed by meal planning tools that include too many advanced features. Simplifying the interface and providing clear navigation can make the process more approachable.

Lack of Personalization

Users want recipes that match their dietary preferences, cooking time, and available ingredients. Providing tailored meal suggestions based on individual inputs can greatly enhance user experience.

Time Constraints

For busy users, the time spent searching for recipes or creating meal plans is a significant pain point. By reducing the number of steps and offering quick, pre-built plans, the app can save time and encourage frequent use.

06 - Deliver

Landing Page

  • Entry Point: Provides a simple landing experience directing users to start exploring, adding, or organizing recipes.
  • ‍ Login Detection: “My Plan” only appears when users are authenticated via Auth0.
  • Visual Tone: Light, welcoming palette and straightforward layout.

All Recipes

  • Recipe Display: Shows curated meals fetched from TheMealDB API via GET requests.
  • ‍ Category Grouping: Recipes are grouped visually by type (e.g., Breakfast, Chicken, Beef), but not filterable via UI controls.
  • Interactive Cards: Clicking on any meal leads to a detail page showing full ingredients and step-by-step instructions.
  • Read-Only Detail View: The instruction page displays content in a two-column layout (ingredients + instructions) with no editing features.

Community Recipes

  • User Submissions : Displays all recipes submitted by users, stored in MongoDB and shown as cards below a hero carousel.
  • ‍ Open Access : Anyone can browse community recipes without logging in.
  • Add New Recipe : Logged-in users can submit a recipe through a form with fields like title, category, ingredients, instructions, and image link. Input validation ensures data completeness.
  • Detailed View : Clicking on a recipe opens a full-page view with the full recipe image, name, category, ingredients, and preparation steps.
  • Edit & Delete Options : Logged-in users see “Edit” and “Delete” buttons. The edit page allows inline editing of recipe fields, which are saved to MongoDB using PUT, while deletion uses DELETE.

My Plan

  • Personal Meal Overview : Logged-in users can view a personalized weekly plan that visually organizes meals under each day of the week.
  • ‍ Combined Data Source : The recipes in the plan come from two merged sources the MealDB API and the MongoDB-based community database offering a mix of curated and user-submitted meals.
  • Image Display Fixes : During development, issues with broken image links from MongoDB were resolved by adjusting how image URLs were stored and rendered.
  • Replace Functionality (Edit) : Each meal card includes an "Edit" button that redirects users to the All Recipes page. Instead of modifying recipe content, users can swap meals by selecting a different one to update their plan.
  • Login-Gated Access : The page is only accessible when logged in via Auth0. Unauthenticated users are redirected to the login page, and dynamic rendering ensures that the “My Plan” option appears only when a session is active.
  • Pagination Debugging : A pagination issue where fewer meals were loaded than expected was resolved by adjusting how recipe limits and variables were handled across pages.

Login & Authentication

  • User Access : Users must sign up or log in using a valid email address and password via Auth0.
  • ‍ Account Options : Supports both manual login and Google-based authentication.
  • Secure Sessions : After authentication, session data is managed securely through tokens to access personalized features like My Plan and Add Recipe.
  • Login Requirement : Pages like My Plan, Add Recipe, and Edit Recipe are only accessible after login.
